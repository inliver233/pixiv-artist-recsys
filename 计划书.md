## 2. 「全量可用」定义（验收标准）

满足以下全部，即视为 **v1 全量可用（个人本地）**：

1. **一键路径**：按 README 用 refresh token 能跑通 `full-recommend`，输出 JSON 含 `user_id/score/reasons/top_illust_ids`
2. **稳定性**：token 刷新失败/429/代理失败有明确错误信息；rotated refresh 不会丢
3. **规模**：支持至少「关注 100+ 画师」的可控运行（有 limit/采样，不会无界打爆 API）
4. **质量门槛**：可配置 AI/R18/min_bookmarks；排除已关注与 dislike/block
5. **可解释**：每条推荐有 reasons + audit 可查询
6. **可复现**：`unittest` 全绿；关键路径有回归测试
7. **文档一致**：README / 计划书 / system-overview 与代码一致
8. **版本管理**：GitHub `main` 为可用基线，`test` 为开发分支

非 v1 目标（明确延后）：Web 完整前端、多账号 SaaS、ML embedding 必选、原图下载站。

## 3. 问题清单（按优先级）

### P0 — 阻塞「敢用」的框架问题

| ID | 问题 | 证据/影响 | 处理策略 |
|----|------|-----------|----------|
| P0-1 | **refresh token 轮换未闭环** | `refresh_into_record` 写 `refresh_token_rotated`，但 `CoordinatorBackedAccessTokenProvider` 始终用调用方传入的旧 refresh；Pixiv 轮换后下次必挂 | 刷新时优先用 rotated；持久化后回读 |
| P0-2 | **实网默认参数过小** | CLI `followed_artist_limit=5` 等导致画像极薄、结果噪声大 | 提升合理默认 + 环境变量 + 文档说明「快速/标准/深度」 |
| P0-3 | **API 爆炸风险** | candidate 对**每个**关注画师打 related；hydrate 对全部关注者拉 illust | 增加 max_seed_artists / 采样策略 |
| P0-4 | **错误不可诊断** | 401/429 只抛通用失败 | 透传 status + 分类消息 |
| P0-5 | **README/架构文档过时** | 仍写「下一步才做鉴权」 | 重写使用说明与阶段 |
| P0-6 | **缺少 .env.example** | 用户不知环境变量 | 补齐示例与安全说明 |

## 7. 风险与约束

1. **Pixiv App API 非官方**：字段/端点可能变更；client 层必须隔离。
2. **限流与风控**：全量 related 会触发 429；必须采样 + backoff。
3. **ToS / 合规**：仅建议个人本地；禁止把 token 提交进 Git。
4. **质量主观**：无法 100% 排除「烂图」；靠门槛 + 反馈迭代。
5. **refresh 轮换**：不落盘新 token = 账号链路自杀；P0-1 不可跳过。
6. **标准库 HTTP**：urllib 够用但弱；后续若稳定性不够再评估 `httpx`。
